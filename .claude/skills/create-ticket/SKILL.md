# create-ticket

GitHub 이슈를 일관된 형식으로 작성·생성한다. 핵심은 **본문을 "작업 내용(무엇을 한다)"과 "목적(왜 필요한가)" 중심으로 쓰는 것**이다. how(구현 단계·체크리스트)나 미정 사항으로 본문이 흐려지지 않게 하고, 추가 입력·결정이 필요한 항목은 본문 맨 아래 "참고 사항" 섹션으로 분리한다.

## 생성 수단

이슈 생성·조회는 **GitHub CLI(`gh`)** 로 한다. 별도 토큰·`.env` 는 필요 없다 — `gh auth login` 으로 인증돼 있으면 된다.

- 생성: `gh issue create`
- 조회: `gh issue view <번호>`
- 인증/대상 레포 확인: `gh auth status`, `gh repo view --json nameWithOwner`

`gh` 가 없거나 인증돼 있지 않으면(`gh auth status` 실패) 사용자에게 `gh auth login` 을 안내하고, 그래도 안 되면 웹에서 수동 생성하도록 안내한다.

## 동작 순서

1. **정보 수집** — 작업 내용(what)과 목적(why)을 파악한다. (대상 레포는 현재 git 레포에서 자동 결정된다.)
2. **본문 작성** — 아래 형식으로 초안을 만든다.
3. **라벨 추천** — 작업 성격에 맞는 라벨을 고른다.
4. **초안 확인** — 제목 / 본문 / 추천 라벨 / 대상 레포를 사용자에게 보여주고 승인을 받는다. (승인 전에는 이슈를 만들지 않는다.)
5. **이슈 생성** — 승인되면 `gh issue create` 로 생성하고, 생성된 이슈 번호·URL 을 돌려준다.
6. **다음 단계 추천** — 이슈 생성 후, `AskUserQuestion` 으로 작업 브랜치를 만들지 묻고, 원하면 `create-branch` 스킬로 이어준다.

## 1. 정보 수집

티켓은 읽는 사람이 **무엇을 하려는지(작업 내용)** 와 **왜 필요한지(목적)** 를 한눈에 파악할 수 있어야 한다. 이 둘이 분명하지 않으면 좋은 티켓이 될 수 없으니, 대화에서 확보되지 않으면 짧게 질문해 채운다.

- **작업 내용(what)**: 무엇을, 어떤 대상·범위로 하는가. 결과물이 무엇인지 구체적으로.
- **목적(why)**: 왜 이 작업이 필요한가. 배경·동기, 해결하려는 문제, 달성하려는 목표.
- **대상 레포**: 어느 GitHub 레포에 만들 것인가. 기본은 현재 작업 디렉토리의 레포다(`gh` 가 `git remote` 에서 자동 결정). 다른 레포에 만들려면 `--repo <owner>/<name>` 으로 지정한다.
- **참고 사항 판단**: 진행에 앞서 **누군가 추가로 입력·결정해야 할 정보**(미정 값, 선택지, 외부 의존성, 현재 진행 현황 등)가 있는가? 있으면 모아 두고, 없으면 이 섹션은 넣지 않는다.

> GitHub 이슈에는 Jira 의 상위/하위(Sub-task) 개념이 없다. 큰 작업을 쪼개야 하면 본문에 GitHub task list(`- [ ] ...`)를 쓰거나 관련 이슈 번호(`#12`)로 참조한다.

## 2. 본문 작성 형식

본문은 다음 두 섹션을 핵심으로 한다.

```markdown
## 작업 내용
<무엇을, 어떤 대상·범위로 하는가. 결과물을 구체적으로. 필요하면 표·목록 사용.>

## 목적
<왜 필요한가. 배경·동기, 해결할 문제, 달성할 목표.>
```

추가 입력·결정이 필요한 항목이 있을 때만 본문 맨 아래에 다음을 덧붙인다. **없으면 통째로 생략한다.**

```markdown
## 참고 사항
<추가 입력·결정이 필요한 항목. 미정 값, 선택지, 의존성, 진행 현황 등.>
```

작성 원칙:
- 헤딩 명칭은 정확히 `작업 내용` / `목적` / `참고 사항`을 쓴다(확정 명칭).
- 체크리스트나 "완료 기준" 같은 how 성격의 상세는 기본적으로 넣지 않는다. 정말 도움이 될 때만 보조적으로 덧붙인다.
- 본문은 한국어로 쓰되, 코드/식별자/명령은 원형 그대로 둔다.
- 리팩토링·성능 작업이면 작업 반영 전 상태를 적는 것이므로 본문에 현재 상태(AS-IS)와 기대 결과(TO-BE)를 함께 적는다.

## 3. 라벨 추천

작업 성격에 맞는 라벨을 1~2개 고른다. 레포마다 라벨 체계가 다르므로, 확신이 없으면 `gh label list` 로 실제 존재하는 라벨을 확인한 뒤 고른다. GitHub 기본 라벨 기준 흔한 매핑:
- 기능 추가 → `enhancement`
- 버그·이슈 → `bug`
- 문서·셋업 → `documentation`

추천 라벨은 초안 확인 단계에서 함께 제시하고, 사용자가 바꾸거나 빼면 그대로 따른다. 레포에 없는 라벨을 지정하면 `gh issue create` 가 실패하므로, 없으면 라벨 없이 진행하거나 `gh label create` 로 먼저 만든다.

## 4. 초안 확인

이슈를 만들기 전에 **항상** 아래를 사용자에게 보여주고 승인을 받는다.

- 제목
- 본문(작업 내용 / 목적 / 필요 시 참고 사항)
- 대상 레포(현재 레포 기본) / 추천 라벨

사용자가 수정을 요청하면 반영해 다시 보여주고, "생성/진행/좋아" 등 명확한 승인이 있을 때만 다음 단계로 넘어간다.

## 5. 이슈 생성

본문에 표·코드블록 등 마크다운이 들어가므로 **본문을 임시 파일에 쓴 뒤 `--body-file` 로 넘긴다.**

```bash
# 본문을 임시 파일로 작성 (Write 도구 사용): /tmp/create-ticket-body.md

gh issue create \
  --title "<제목>" \
  --body-file /tmp/create-ticket-body.md \
  [--label enhancement] \
  [--repo <owner>/<name>]
```

- 라벨은 `--label` 을 여러 번(또는 콤마로) 넘긴다. 레포에 존재하는 라벨만 받는다.
- 다른 레포에 만들 때만 `--repo <owner>/<name>` 을 쓴다. 보통은 생략(현재 레포).
- `gh issue create` 는 성공 시 생성된 이슈 URL 을 출력한다. URL 끝의 번호가 이슈 번호(`#42`)다. 이 번호와 URL 을 사용자에게 전달한다.

## 6. 다음 단계 추천 (AskUserQuestion)

이슈를 만들었으면 **`AskUserQuestion` 도구로 작업 브랜치를 만들지 묻는다.** 바로 작업을 시작할 때는 편하지만 늘 원하는 건 아니므로, 임의로 만들지 말고 물어본다.

- 질문: "방금 만든 이슈(`#<번호>`)로 작업 브랜치를 생성할까요?"
- 헤더: `다음 단계`
- 선택지:
  - **브랜치 생성 (create-branch)** — 베이스 브랜치(예: `main`)에서 `<type>/<이슈번호>-<slug>` 형식의 브랜치를 만들어 체크아웃한다.
  - **아니요** — 여기서 멈춘다.

사용자가 "브랜치 생성"을 고르면 `create-branch` 스킬로 이어준다 — 베이스 브랜치와 방금 만든 이슈 번호를 넘긴다.

- 이 스킬에서 직접 브랜치를 만들지 않는다. 브랜치 생성 책임은 `create-branch` 가 단독으로 갖는다.

## 예시

**제목:** 이미지 검색 API 연동

```markdown
## 작업 내용
검색어를 입력하면 외부 이미지 검색 API를 호출해 결과 이미지 목록을 받아온다.
`data → domain → presentation` 계층 순으로 연동하고, 결과를 메인 화면에 그리드로 표시한다.

## 목적
현재 앱은 플레이스홀더만 렌더링하는 스캐폴드 상태로 핵심 가치인 "이미지 검색"이 동작하지 않는다.
사용자가 실제로 검색하고 결과를 볼 수 있어야 이후 기능(상세 보기, 저장 등)을 쌓을 수 있다.

## 참고 사항
- 사용할 검색 API(Unsplash / Pixabay / 자체 백엔드) 미정
- API Key 보관 위치(local.properties vs CI 시크릿) 결정 필요
```

이 예시에서 추가 입력·결정이 필요한 항목이 없었다면 `## 참고 사항`은 넣지 않는다.

이 이슈가 `#42` 로 생성됐다면, 6단계에서 `create-branch` 로 제안할 브랜치는 베이스(예: `main`) 기준 `feature/42-image-search-api` 다.
