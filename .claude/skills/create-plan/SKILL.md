# Create Plan

비자명한 작업을 **코드로 검증된 구현 계획**으로 만들고, 그 계획을 **혼자서도 완결 가능한 독립 task** 들로 분해해 Claude Code Task 시스템에 등록하는 스킬.

## 왜 이 스킬이 필요한가

구현을 시작하기 전 계획을 세울 때 흔히 두 가지가 어긋난다:

- **추측으로 계획을 적는다.** 코드를 보지 않고 "이 화면을 Compose 로 바꾸면 된다" 식으로 적으면, 실제로는 이미 같은 패턴을 구현한 선례가 있거나(재발명 낭비), 건드리면 안 되는 의존이 숨어 있어 계획이 첫 구현에서 깨진다.
- **task 가 서로 얽혀 있다.** "1단계 다 하고 2단계" 식으로 뭉뚱그리면, 각 task 만 보고는 무엇을 어디까지 해야 하는지 알 수 없다. 범위가 새고, 검증 기준이 없고, 누가(다음 세션의 나 포함) 이어받아도 헤맨다.

이 스킬은 (1) 계획을 적기 전에 **코드와 기존 선례를 먼저 조사**해 사실에 근거하게 하고, (2) 계획을 **파일 단위로 구체화**해 실제 작업량과 위험을 드러내며, (3) 결과를 **각자 자기완결적인 독립 task** 로 분해해 등록한다.

## 산출물 — 무엇을 만드는가

두 가지를 함께 낸다:

1. **계획 제시** (대화에 출력) — 전략 한 줄 + 신규/수정/삭제 **파일 표** + "변경 없음" 경계 + 확인이 필요한 결정.
2. **등록된 task** (`TaskCreate`) — 독립적이고 자기완결적인 task 들. 실제 실행 순서 의존이 있으면 `blockedBy` 로 명시.

task 는 **Claude Code 세션의 Task 시스템**(`TaskCreate`/`TaskUpdate`/`TaskList`)에 등록한다. Jira 티켓이나 별도 문서가 아니다 — 이 스킬의 task 는 이번 작업을 추적하는 세션 내 작업목록이다.

## 핵심 원칙

### 1. 계획을 적기 전에 코드로 조사한다

티켓·스펙·관련 코드를 직접 읽는다. 스펙(`.claude/docs/specs/`)이나 기준 문서가 있으면 그것을 task 의 **acceptance criteria 출처**로 삼는다.

특히 **코드베이스에서 기존 선례를 찾는다** — 지금 하려는 것과 비슷한 기능이 이미 구현돼 있나. 있으면 그 패턴을 재사용하는 게 새로 발명하는 것보다 빠르고 안전하다. (예: "보관함 아이템을 Compose 로 전환" 계획에서, 같은 콤보 셀을 이미 Compose 로 구현한 `donate/sticker` 를 찾아 그 구조를 그대로 따른다.) `grep`/`Explore` 로 유사 화면·유사 컴포넌트·유사 시그니처를 훑어 선례를 확정한다.

조사 없이 적은 계획은 거의 항상 틀린다. 시간을 들여도 좋으니 여기서 사실을 확보한다.

### 2. 파일 단위로 구체화한다

좋은 계획은 전략을 한 줄로 정하고, 그다음 **정확히 어떤 파일이 신규/수정/삭제되는지** 표로 적는다. 그리고 **"변경 없음"** 경계를 명시해, 읽는 사람이 "이건 일부러 안 건드린다"를 알게 한다.

"셀을 마이그레이션한다" 같은 뭉뚱그린 표현은 실제 작업과 위험을 숨긴다. 파일 표는 작업량을 정직하게 드러내고, 그대로 task 분해의 기반이 된다.

### 3. 정할 수 있는 건 정하고, 진짜 갈림길만 묻는다

코드·컨벤션·선례로 결정할 수 있는 것은 스스로 정하고 계획에 적는다("X 로 한다"). 사용자에게는 **결과를 실제로 바꾸는 진짜 양자택일**, 그리고 코드만으로 단정할 수 없는 것만 `AskUserQuestion` 으로 묻는다(예: "호스트의 빈/로딩 표시까지 Compose 로 흡수할지 vs 그리드만 교체할지").

자명한 기본값이 있는 것을 묻는 것은 사용자의 시간을 낭비한다. 추천안을 첫 옵션으로 두고 근거를 함께 제시한다.

### 4. 독립적이고 자기완결적인 task 로 분해한다 (이 스킬의 핵심)

각 task 는 **그 설명만 읽고도 단독 수행이 가능**해야 한다. task 하나에 다음이 모두 들어간다:

- **범위** — 이 task 가 끝내는 정확한 한 조각 (작업 위치/경로 포함)
- **기준 문서** — 어떤 스펙/문서의 어느 부분이 이 task 의 acceptance criteria 인가
- **파일** — 신규/수정/삭제할 구체 파일 경로
- **선례** — 따를 기존 코드 (있으면)
- **변경 없음** — 이 task 가 건드리지 않는 것
- **검증/완료기준** — 이 task 가 끝났음을 *어떻게* 확인하나 (Preview, 빌드, 체크리스트 대조 등)

이렇게 쓰면 task 는 **자기완결적 수직 슬라이스** — 고유 산출물과 고유 검증을 가진 한 단위 — 가 된다.

**"독립적"의 의미를 정직하게 다룬다.** 자기완결(self-contained)과 병렬 독립(parallel-independent)은 다르다. 마이그레이션·리팩토링은 보통 셀→그리드→통합→정리처럼 실제 순서 의존이 있다. 이때 task 를 억지로 "완전 병렬"인 척 적지 말고, `blockedBy` 로 의존을 솔직히 표시한다. 각 task 는 자기완결적이되, 실행 순서는 의존 사슬로 강제하는 것이 맞다.

### 5. 계획을 먼저 제시하고, 확정 후 등록한다

`TaskCreate` 로 바로 등록하지 말고, 먼저 계획(전략 + 파일 표 + 결정)을 사용자에게 제시한다. 진짜 갈림길(원칙 3)을 해소하고 계획 모양이 합의되면, 그때 task 를 등록한다. 계획이 흔들리는데 task 부터 박아 넣으면 되돌리는 비용이 크다.

## 절차

1. **작업 이해** — 입력(티켓 키/스펙/요구)을 확인하고 관련 코드를 읽는다. 기준 문서가 있으면 acceptance criteria 출처로 잡는다.
2. **선례 리서치** — `grep`/`Explore` 로 유사 구현을 찾아 재사용할 패턴을 확정한다. 빌드 산출물(`build/`)은 무시하고 `src/` 소스만 본다.
3. **전략 결정** — 방향을 한 줄로 정한다(survey 가 아니라 추천 하나). 큰 UI 개편이면 "기존 미수정 + 신규 병렬 구현" 같은 프로젝트 컨벤션을 따른다.
4. **파일 단위 구체화** — 신규/수정/삭제 파일 표 + "변경 없음" 경계.
5. **진짜 결정 질문** — `AskUserQuestion` 으로 실제 갈림길만. 정할 수 있는 건 정해서 적는다.
6. **task 분해 + 등록** — 자기완결 task 로 쪼개 `TaskCreate` 로 등록하고, 의존이 있으면 `TaskUpdate` 의 `addBlockedBy` 로 순서를 건다. 등록 전 기존 `TaskList` 를 확인해 중복을 피한다.
7. **독립성 자체점검** — 아래 점검을 돌리고, 결과를 사용자에게 보고한다.

## task 자체점검 — 등록 전/후 확인

각 task 에 대해 스스로 묻는다:

- [ ] 이 task **설명만** 읽고 다른 맥락 없이 수행할 수 있나? (범위·파일·기준·검증이 다 들어 있나)
- [ ] 이 task 만의 **완료기준과 검증 방법**이 있나? ("빌드 통과", "Preview 확인", "스펙 체크리스트 대조" 등 구체적으로)
- [ ] task 끼리 범위가 **겹치거나 새지** 않나? (한 task 의 산출물이 다른 task 로 흘러넘치지 않나)
- [ ] 실제 순서 의존을 `blockedBy` 로 정직히 표시했나? (병렬인 척하지 않았나)
- [ ] "변경 없음" 경계가 task 마다 명확한가?

하나라도 아니면 그 task 를 다시 쓴다. 자기완결이 안 되는 task 는 분해를 잘못한 것이다.

## 넣지 않는 것 / 안티패턴

- **추측 기반 계획** — 코드를 안 보고 적은 단계. 원칙 1 위반.
- **뭉뚱그린 task** — "UI 작업", "통합" 처럼 파일·기준·검증이 없는 한 줄. 받는 사람이 헤맨다.
- **결정 가능한 것을 묻기** — 자명한 기본값이 있는데 사용자에게 떠넘기기.
- **거짓 병렬** — 순서 의존이 있는데 `blockedBy` 없이 독립인 척.
- **계획 없이 task 부터 등록** — 합의 전 `TaskCreate` 남발.
- **구현까지 진행** — 이 스킬은 계획·등록까지다. 구현은 사용자가 task 를 집어 진행한다(별도 흐름).

## 좋은 계획의 신호

- 파일 표를 보면 작업량과 위험이 한눈에 보인다.
- 각 task 가 그 자체로 작은 명세 — 범위·파일·기준·검증을 갖춘다.
- "변경 없음"이 명시돼 오해를 막는다.
- 선례가 명시돼 재구현자가 무엇을 따라야 할지 안다.
- 의존 사슬이 `blockedBy` 로 정직하게 드러난다.
- 사용자가 답한 결정이 계획에 반영돼 있다.
