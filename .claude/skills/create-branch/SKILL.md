# Create Branch

베이스 브랜치에서 GitHub 워크플로에 맞는 이름으로 새 브랜치를 생성하고 체크아웃하는 스킬.

## 브랜치 네이밍 규칙

```
<type>/<이슈번호>-<slug>
```

- **`<type>`** — 작업 성격. `feature` | `fix` | `refactor` | `docs` | `chore` | `test` 중 하나. 커밋 타입·PR Type 과 같은 어휘를 쓴다.
- **`<이슈번호>`** — 연결된 GitHub 이슈 번호(`#` 없이 숫자만). 연결 이슈가 없으면 이 부분을 생략하고 `<type>/<slug>` 로 만든다.
- **`<slug>`** — 작업을 요약한 kebab-case 영문 슬러그. 2~5단어, 동작 중심(예: `add-ktor-adapter`, `fix-cache-key`).

**예시:**
- `feature/42-image-search-api` (이슈 #42 연결)
- `fix/57-ktor-mock-delay`
- `chore/bump-compose-bom` (연결 이슈 없음)

## 필요한 인자

1. **베이스 브랜치명** — 분기의 기준이 되는 브랜치 (예: `main`, `develop`)
2. **브랜치명** — 위 규칙으로 조립한 새 브랜치 이름

## 사용법

번들된 스크립트를 실행한다:

```bash
python3 <skill-directory>/scripts/create_branch.py <베이스_브랜치> <브랜치명>
```

**예시:**
```bash
python3 <skill-directory>/scripts/create_branch.py main feature/42-image-search-api
```

## 베이스 브랜치 결정

- **대화 중에 베이스 브랜치를 파악했으면, 그 베이스 브랜치에서 새 브랜치를 생성한다.** (예: 직전 대화에서 `develop`에서 작업한다고 했으면 그 브랜치를 베이스로 쓴다.)
- 대화에서 베이스 브랜치를 파악하기 어려우면 사용자에게 베이스 브랜치를 물어본다. 임의로 `develop`/`main` 등을 가정하지 않는다.

## 브랜치명 조립

- **`<type>`**: 작업 성격에서 고른다(`feature`/`fix`/`refactor`/`docs`/`chore`/`test`). 애매하면 사용자에게 확인한다.
- **`<이슈번호>`**: 대화 맥락(직전에 만든 이슈 등)에서 파악했으면 그대로 쓴다. `create-ticket` 에서 위임돼 왔으면 그 이슈 번호를 쓴다. 연결할 이슈가 없으면 생략한다.
- **`<slug>`**: 작업 내용을 2~5단어 kebab-case 영문으로 요약한다.

## 동작 순서

1. `git fetch origin <베이스_브랜치>` — 원격의 베이스 브랜치를 최신 상태로 가져온다
2. `git checkout -b <브랜치명> --no-track origin/<베이스_브랜치>` — 베이스 브랜치 기준으로 새 브랜치를 생성하고 체크아웃한다

## 주의사항

- Git 저장소 내에서 실행해야 한다. 저장소 밖에서 실행하면 오류가 발생한다.
- 이미 동일한 이름의 브랜치가 존재하면 git이 오류를 반환한다. 이 경우 사용자에게 알려주고 기존 브랜치를 체크아웃할지 확인한다.
- 원격에 해당 베이스 브랜치가 없으면 fetch가 실패한다. 브랜치명을 다시 확인하도록 안내한다.

## 인자가 누락된 경우

두 인자(베이스 브랜치, 브랜치명) 중 누락된 것을 채운다.

- **베이스 브랜치**: 대화 맥락에서 파악했으면 그대로 쓰고, 파악이 어려우면 사용자에게 물어본다.
- **브랜치명**: 대화 맥락(직전에 만든 이슈, 작업 내용)에서 type·이슈번호·slug 를 조립한다. 부족하면 물어본다.

## 다음 단계 — 어디서 작업할지

브랜치를 만든 뒤, **`AskUserQuestion` 으로 "어디서 작업할지"를 묻는다.**

- **메인 트리에서 작업** — 방금 체크아웃한 브랜치 그대로 작업을 시작한다(추가 동작 없음).
- **새 worktree 에서 작업** — `create-worktree` 로 넘겨 별도 작업 공간을 만든다.

같은 브랜치를 두 곳에 체크아웃할 수 없으므로 이 분기는 브랜치 생성 직후에 둔다.
