# create-commit

작업을 마무리할 때 두 가지를 한 흐름으로 처리한다: ① 변경 사항을 논리적 단위로 분류해
커밋하고, ② 그 작업에서 지식 베이스에 남길 가치가 있는 것이 생겼는지 점검해 `knowledge-update`로
이어준다.

**왜 커밋 직후인가.** 결정의 배경, 이슈의 원인, 새로 합의한 규칙 같은 지식은 "남길 만한
순간"에 바로 적지 않으면 잊힌다. 커밋은 한 덩어리의 작업이 끝나는 자연스러운 체크포인트라,
여기서 한 번 돌아보면 지식이 유실되지 않는다.

## 1. 변경 사항 파악

`git status`와 `git diff`(스테이지 포함 `--staged`)로 무엇이 바뀌었는지 본다. 커밋 메시지와
분류는 추측이 아니라 실제 변경 내용에 근거해야 한다.

추적되지 않은 파일 중 **내가 만들지 않은 것**(설치된 도구, 대용량 산출물, 벤더 디렉토리 등)이
섞여 있으면, 분류하기 전에 사용자에게 커밋에 포함할지 확인한다. 의도치 않게 커밋하면 히스토리에
남아 되돌리기 번거롭기 때문이다.

## 2. 브랜치 안전장치

현재 브랜치가 기본 브랜치(보통 `main`/`master`)이면 커밋하기 전에 작업 브랜치를 만든다.
기본 브랜치에 직접 커밋을 쌓지 않는다.

## 3. 분류

서로 다른 관심사는 별도 커밋으로 나눈다. 한 커밋은 하나의 일관된 변경이어야 나중에 읽고
되돌리기 쉽다. 예: 기능 추가 / 버그 수정 / 문서 / 설정·도구는 각각 분리.

### 커밋 메시지 형식

**제목**: `<타입>: <요약>` (conventional commits 스타일)

- `<타입>` — 변경 성격을 `feat` / `fix` / `docs` / `refactor` / `test` / `chore`
  중에서 고른다.
- `<요약>` — 이 커밋이 무엇을 하는지 한 줄로. 명령형·소문자로 시작하고 마침표는 붙이지 않는다.
- **이슈 연결(선택)** — 연결된 GitHub 이슈가 있으면 제목 끝에 `(#<번호>)` 를 붙인다. 이슈 번호는
  현재 브랜치명(`<type>/<이슈번호>-<slug>`)에서 뽑는다. 없으면 생략한다. (이슈 자동 종료는 보통
  PR 본문의 `Closes #<번호>` 에서 처리하므로, 커밋 제목의 `(#N)` 는 참조용이다.)
- 예: `feat: add social login button to login screen (#42)`
- 예: `test: add core-logic unit tests and JaCoCo coverage`

**메시지(본문)**: `<변경 사항>` — 이번 커밋에서 실제로 무엇을 바꿨는지 적는다. 변경이 자명하면
생략하거나 한 줄로 충분하다.

- 환경에서 요구하는 트레일러(예: `Co-Authored-By`)가 있으면 포함한다.

분류가 애매하거나 한 파일에 여러 관심사가 섞여 있으면 사용자에게 어떻게 나눌지 확인한다.

## 4. 커밋

분류한 단위대로 `git add <경로>` → `git commit` 한다. 폴더 단위로 명시적으로 stage 해서
의도한 파일만 들어가게 한다(`git add .` 로 무심코 다른 변경을 끌어오지 않도록). 끝나면
`git log --oneline`과 `git status`로 결과를 사용자에게 보고한다.

푸시는 사용자가 요청할 때만 한다. 커밋까지 마치고 나면 푸시할지 물어볼 수 있다.

## 5. 지식 점검 (핵심)

방금 커밋한 작업과 이번 세션의 대화를, 지식 베이스의 "기록 시점"에 비춰 돌아본다:

| 카테고리 | 이번 작업에 이런 게 있었나? |
|---|---|
| `decisions` | 기술·라이브러리·구조·방향을 선택했거나, 대안을 검토하고 버렸나 |
| `troubleshootings` | 버그·빌드·CI 실패를 원인까지 파악해 해결했나 |
| `domains` | 프로젝트에서만 통하는 용어를 새로 정의·정리했나 |
| `conventions` | 코드·테스트·스타일 규칙을 정하거나 합의했나 |
| `flows` | 특정 종류의 작업을 개발하는 순서·절차를 정립했나 |

**판단 기준은 "재사용 가능하고 비자명한가"이다.** 일상적인 기능 코드, 포맷팅, 단순 의존성
갱신, 오타 수정 등은 대개 기록 대상이 아니다. 지식 베이스를 노이즈로 채우면 정작 중요한 것이
묻히므로, 다음에 또 마주칠 사람이 시간을 아낄 수 있는 것만 후보로 본다.

## 6. 다음 단계 추천 (AskUserQuestion)

커밋을 마치면 **`AskUserQuestion` 도구로 다음 단계를 묻는다.** 선택지는 5단계 지식 점검
결과에 따라 달라지며, 지식 기록과 리뷰는 함께 할 수 있으므로 `multiSelect: true` 로 묻는다.

- 질문: "커밋을 마쳤어요. 다음으로 무엇을 할까요?"
- 헤더: `다음 단계`
- `multiSelect`: true
- 선택지:
  - **커밋 리뷰 (review-commit)** — base 브랜치 기준으로 이번 브랜치 커밋을 3관점으로
    리뷰한다. 고르면 `review-commit` 으로 이어주되, base 브랜치를 함께 확인한다(인자 필수 —
    자동 추정하지 않는다). 리뷰에서 지적이 나오면 수정 후 다시 이 스킬(create-commit)로
    돌아와 수정분을 커밋하게 되므로, 보통 단독으로 고른다.
  - **지식 기록 (knowledge-update)** — *5단계에서 남길 만한 항목이 있을 때만 이 선택지를
    넣는다.* 어느 카테고리에 무엇을 남길지 한 줄 요약을 description 에 적는다. 고르면
    `knowledge-update` 로 이어준다(이 스킬은 기록 후 `knowledge-drift` 정합성 점검까지 필수로 수행한다).
  - **PR 생성 (create-pr)** — 현재 브랜치에서 base 브랜치로 향하는 PR을 만든다. 고르면
    `create-pr` 로 이어준다.
  - **끝내기** — 여기서 멈춘다.

- **5단계에서 남길 지식이 있으면**: "지식 기록" 선택지를 넣고 한 줄 요약을 곁들인다.
- **없으면**: "지식 기록" 선택지를 빼고, "이번 커밋에서 지식 베이스에 추가할 만한 항목은
  없어 보인다"고 짧게 알린 뒤 나머지 선택지로 묻는다.
- 여러 개를 고르면 `knowledge-update`(+필수 drift) → `create-pr` 순으로 진행한다.
- 여기서 직접 항목을 만들거나 리뷰를 임의로 시작하지 않는다 — 무엇을 할지는 사용자가 고른다.

## 참고

이번 커밋이 `.claude/docs/`의 항목·인덱스를 건드렸다면, `knowledge-drift` 스킬로 인덱스
정합성도 함께 점검할 수 있다. `knowledge-update`로 기록하고, `knowledge-drift`로 점검하는
짝과 함께 쓰면 지식 베이스가 일관되게 유지된다.
