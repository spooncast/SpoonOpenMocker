# create-ticket

Jira 이슈를 일관된 형식으로 작성·생성한다. 핵심은 **본문을 "작업 내용(무엇을 한다)"과 "목적(왜 필요한가)" 중심으로 쓰는 것**이다. how(구현 단계·체크리스트)나 미정 사항으로 본문이 흐려지지 않게 하고, 추가 입력·결정이 필요한 항목은 본문 맨 아래 "참고 사항" 섹션으로 분리한다.

## 생성 수단 (우선순위)

티켓 생성·조회는 아래 순서로 수단을 고른다. **항상 1번을 먼저 시도하고, 불가할 때만 2번으로 내려간다.**

1. **dexus MCP** (연결돼 있으면 우선) — `mcp__dexus__jira_issue_manage` 를 사용한다. 도구 스키마는 `ToolSearch` 로 `select:mcp__dexus__jira_issue_manage` 를 불러와 호출한다. 생성은 `action=create`, 조회는 `action=get`.
2. **Jira REST API 스크립트** (dexus 미연결 시 fallback) — `scripts/create_jira_issue.py` 로 생성한다. `.env` 의 `JIRA_BASE_URL` / `JIRA_EMAIL` / `JIRA_API_TOKEN` 이 필요하다. 이마저 없으면 사용자에게 수동 생성을 안내한다.

> dexus 연결 여부는 `mcp__dexus__*` 도구가 호출 가능한지로 판단한다. 확신이 안 서면 `ToolSearch` 로 `select:mcp__dexus__jira_issue_manage` 를 시도해보고, 로드되면 dexus 경로, 실패하면 스크립트 경로로 진행한다.

## 동작 순서

1. **정보 수집** — 작업 내용(what)과 목적(why), 그리고 프로젝트 키·상위 티켓을 파악한다.
2. **본문 작성** — 아래 형식으로 초안을 만든다.
3. **라벨/컴포넌트 추천** — 작업 성격에 맞는 라벨을 고른다.
4. **초안 확인** — 제목 / 본문 / 추천 라벨 / 생성 수단을 사용자에게 보여주고 승인을 받는다. (승인 전에는 이슈를 만들지 않는다.)
5. **이슈 생성** — 승인되면 dexus 또는 스크립트로 생성하고, 생성된 이슈 키·URL 을 돌려준다.
6. **다음 단계 추천** — 이슈 생성 후, `AskUserQuestion` 으로 작업 브랜치를 만들지 묻고, 원하면 `create-branch` 스킬로 이어준다.

## 1. 정보 수집

티켓은 읽는 사람이 **무엇을 하려는지(작업 내용)** 와 **왜 필요한지(목적)** 를 한눈에 파악할 수 있어야 한다. 이 둘이 분명하지 않으면 좋은 티켓이 될 수 없으니, 대화에서 확보되지 않으면 짧게 질문해 채운다.

- **작업 내용(what)**: 무엇을, 어떤 대상·범위로 하는가. 결과물이 무엇인지 구체적으로.
- **목적(why)**: 왜 이 작업이 필요한가. 배경·동기, 해결하려는 문제, 달성하려는 목표.
- **프로젝트 키**: 어느 Jira 프로젝트에 만들 것인가(예: `AOSTEAM`). 현재 git 브랜치명의 티켓 키(`[A-Z]+-[0-9]+`) prefix 에서 추정하고, 추정되면 사용자에게 확인한다. 없으면 묻는다.
- **상위 티켓**: **대화 중에 상위 티켓을 파악했으면, 그 상위 티켓의 하위 태스크(Sub-task)로 만든다.** 대화에서 상위 티켓을 파악하기 어려우면 사용자에게 상위 티켓을 물어본다(브랜치명 티켓 키를 상위 후보로 제안할 수 있다). 사용자가 상위 티켓이 없다고 하면 일반 이슈(`Task`)로 만든다.
- **참고 사항 판단**: 진행에 앞서 **누군가 추가로 입력·결정해야 할 정보**(미정 값, 선택지, 외부 의존성, 현재 진행 현황 등)가 있는가? 있으면 모아 두고, 없으면 이 섹션은 넣지 않는다.

## 2. 본문 작성 형식

본문은 다음 두 섹션을 핵심으로 한다.

```markdown
## 작업 내용
<무엇을, 어떤 대상·범위로 하는가. 결과물을 구체적으로. 필요하면 표·목록 사용.>

## 목적
<왜 필요한가. 배경·동기, 해결할 문제, 달성할 목표.>
```

추가 입력·결정이 필요한 항목이 있을 때만 본문 맨 아래에 다음을 덧붙인다. **없으면 통째로 생략한다.**

```markdown
## 참고 사항
<추가 입력·결정이 필요한 항목. 미정 값, 선택지, 의존성, 진행 현황 등.>
```

작성 원칙:
- 헤딩 명칭은 정확히 `작업 내용` / `목적` / `참고 사항`을 쓴다(사용자 확정 명칭).
- 체크리스트나 "완료 기준" 같은 how 성격의 상세는 기본적으로 넣지 않는다. 정말 도움이 될 때만 보조적으로 덧붙인다.
- 본문은 한국어로 쓰되, 코드/식별자/명령은 원형 그대로 둔다.
- 리팩토링·성능 작업이면 작업 반영 전 상태를 적는 것이므로 본문에 현재 상태(AS-IS)와 기대 결과(TO-BE)를 함께 적는다.

## 3. 라벨/컴포넌트 추천

작업 성격에 맞는 라벨을 1~2개 고른다. Jira 프로젝트마다 라벨·컴포넌트 체계가 다르므로, 확신이 없으면 dexus(`mcp__dexus__jira_project_manage`)나 기존 유사 티켓에서 실제 사용되는 라벨을 확인한 뒤 고른다. 흔한 매핑:
- 기능 추가 → `feature` / `enhancement`
- 버그·이슈 → `bug`
- 문서·셋업 → `docs`

추천 라벨은 초안 확인 단계에서 함께 제시하고, 사용자가 바꾸거나 빼면 그대로 따른다. 마땅한 라벨이 없으면 라벨 없이 진행한다.

## 4. 초안 확인

이슈를 만들기 전에 **항상** 아래를 사용자에게 보여주고 승인을 받는다.

- 제목
- 본문(작업 내용 / 목적 / 필요 시 참고 사항)
- 프로젝트 키 / 상위 티켓(있으면) / 이슈 타입
- 추천 라벨
- 생성 수단(dexus / 스크립트)

사용자가 수정을 요청하면 반영해 다시 보여주고, "생성/진행/좋아" 등 명확한 승인이 있을 때만 다음 단계로 넘어간다.

## 5. 이슈 생성

### 5-A. dexus MCP (우선)

`ToolSearch` 로 `select:mcp__dexus__jira_issue_manage` 를 불러온 뒤 `action=create` 로 호출한다. 전달 필드:
- 프로젝트 키, 이슈 타입(`Task` 또는 하위 태스크면 `Sub-task`), 제목(summary)
- 본문(description) — "작업 내용 / 목적 / 참고 사항"
- 상위 티켓(parent, 하위 태스크일 때), 라벨/컴포넌트(있으면)

생성 후 반환된 이슈 키와 브라우저 URL 을 사용자에게 전달한다.

### 5-B. 스크립트 (fallback)

본문에 표·코드블록 등 마크다운이 들어가므로 **본문을 임시 파일에 쓴 뒤 `--description-file` 로 넘긴다.**

```bash
# 본문을 임시 파일로 작성 (Write 도구 사용): /tmp/create-ticket-body.md

python3 <skill-directory>/scripts/create_jira_issue.py \
  --project AOSTEAM \
  --summary "<제목>" \
  --description-file /tmp/create-ticket-body.md \
  --type Task \
  [--parent AOSTEAM-1234] \
  [--labels feature android] \
  [--components network]
```

- 하위 태스크로 만들 때만 `--parent` 와 `--type Sub-task` 를 함께 쓴다.
- `.env` 에 `JIRA_BASE_URL` / `JIRA_EMAIL` / `JIRA_API_TOKEN` 이 없으면 스크립트가 안내하며 종료한다 — 이때는 사용자에게 수동 생성을 안내한다.
- 생성 후 출력되는 이슈 키·URL 을 사용자에게 전달한다.

## 6. 다음 단계 추천 (AskUserQuestion)

이슈를 만들었으면 **`AskUserQuestion` 도구로 작업 브랜치를 만들지 묻는다.** 바로 작업을 시작할 때는 편하지만 늘 원하는 건 아니므로, 임의로 만들지 말고 물어본다.

- 질문: "방금 만든 티켓(`<티켓 키>`)으로 작업 브랜치를 생성할까요?"
- 헤더: `다음 단계`
- 선택지:
  - **브랜치 생성 (create-branch)** — 베이스 브랜치(예: `release/x.y.z`, `develop`)에서 티켓 키 이름의 브랜치를 만들어 체크아웃한다.
  - **아니요** — 여기서 멈춘다.

사용자가 "브랜치 생성"을 고르면 `create-branch` 스킬로 이어준다 — 베이스 브랜치와 방금 만든 Jira 티켓 키를 넘긴다.

- 이 스킬에서 직접 브랜치를 만들지 않는다. 브랜치 생성 책임은 `create-branch` 가 단독으로 갖는다.

## 예시

**제목:** 이미지 검색 API 연동

```markdown
## 작업 내용
검색어를 입력하면 외부 이미지 검색 API를 호출해 결과 이미지 목록을 받아온다.
`data → domain → presentation` 계층 순으로 연동하고, 결과를 메인 화면에 그리드로 표시한다.

## 목적
현재 앱은 플레이스홀더만 렌더링하는 스캐폴드 상태로 핵심 가치인 "이미지 검색"이 동작하지 않는다.
사용자가 실제로 검색하고 결과를 볼 수 있어야 이후 기능(상세 보기, 저장 등)을 쌓을 수 있다.

## 참고 사항
- 사용할 검색 API(Unsplash / Pixabay / 자체 백엔드) 미정
- API Key 보관 위치(local.properties vs CI 시크릿) 결정 필요
```

이 예시에서 추가 입력·결정이 필요한 항목이 없었다면 `## 참고 사항`은 넣지 않는다.

이 이슈가 `AOSTEAM-1500` 으로 생성됐다면, 6단계에서 `create-branch` 로 제안할 브랜치는 베이스(예: `develop`) 기준 `AOSTEAM-1500` 이다.
