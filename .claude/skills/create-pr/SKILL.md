# create-pr

현재 브랜치를 베이스 브랜치로 머지하기 위한 GitHub PR을 일관된 형식으로 작성·생성한다. 핵심은 **본문을 정해진 템플릿(Meta·문제/신규 기능 설명·적용 버전·원인·수정/구현)으로 채우는 것**이다.

**왜 이 구성인가.** PR을 읽는 동료는 보통 시간에 쫓긴다. 맨 위 PR Type 으로 변경 성격(기능/리팩토링/버그/크래시)을 즉시 알리고, *문제 → 원인 → 수정* 흐름으로 *무엇을 왜 어떻게 바꿨는지*를 항상 같은 자리에서 찾게 하면 리뷰 피로가 줄고 머지가 빨라진다. 그래서 매 PR이 동일한 템플릿을 따른다.

## 생성 수단

PR 생성과 연결 이슈 조회는 **GitHub CLI(`gh`)** 로 한다. 별도 토큰·`.env` 는 필요 없다 — `gh auth login` 으로 인증돼 있으면 된다.

- PR 생성: `gh pr create`
- 같은 브랜치의 기존 PR 조회: `gh pr list --head <브랜치>` / `gh pr view`
- 연결 이슈 조회: `gh issue view <번호> --json title,body`

`gh` 가 없거나 인증돼 있지 않으면(`gh auth status` 실패) 사용자에게 `gh auth login` 을 안내한다.

## 동작 순서

1. **상태 파악** — 현재 브랜치, 베이스 브랜치, 이 브랜치의 커밋·변경 파일, 원격 추적 상태를 확인한다.
2. **기존 PR 확인** — 같은 브랜치의 열린 PR이 있으면 중복 생성하지 않는다.
3. **연결 이슈 식별** — 브랜치명에서 이슈 번호를 뽑아 해당 이슈의 제목·본문을 가져온다.
4. **본문 작성** — 아래 템플릿 형식으로 초안을 만든다.
5. **초안 확인** — 제목 / 본문 / 베이스 브랜치를 사용자에게 보여주고 승인을 받는다. (승인 전에는 푸시·PR 생성을 하지 않는다.)
6. **PR 생성** — 승인되면 필요 시 브랜치를 푸시한 뒤 `gh pr create` 로 생성하고, PR URL을 돌려준다.

## 1. 상태 파악

PR 본문은 추측이 아니라 **실제 변경**에 근거해야 한다. 먼저 무엇이 바뀌었는지 본다.

먼저 PR을 올릴 **타겟(베이스) 브랜치**를 정한다.

- **대화 중에 타겟 브랜치를 파악했으면, 그 타겟 브랜치로 PR을 생성한다.** (예: 직전 대화에서 `develop`로 머지한다고 했으면 그 브랜치를 베이스로 쓴다.)
- 대화에서 파악이 어려우면 아래 순서로 추정하고, 그래도 못 찾으면 사용자에게 물어본다.

```bash
BRANCH=$(git branch --show-current)

# 레포의 기본 브랜치 확인 (베이스 후보)
gh repo view --json defaultBranchRef --jq .defaultBranchRef.name

# 베이스(분기 시점) 브랜치 추정 — reflog 에서 분기 기록을 찾는다
git reflog show --no-abbrev HEAD | grep "branch: Created from"
```

- **대화 맥락이나 reflog·기본 브랜치로 베이스를 찾은 경우**: 해당 브랜치를 베이스로 사용한다.
- **못 찾은 경우**: 사용자에게 묻는다 — "타겟 브랜치를 자동으로 찾지 못했어요. 어떤 브랜치로 PR을 올릴까요? (예: main, develop)"

베이스(`BASE`)가 정해지면 변경을 분석한다.

```bash
git log "$BASE"..HEAD --oneline      # 이 브랜치가 베이스보다 앞선 커밋
git diff "$BASE"...HEAD --stat       # 변경 파일 요약 (3-dot: 분기 이후 변경만)
git status -sb                       # 원격 추적·동기화(ahead/behind) 상태
```

- `$BASE`와 차이가 없으면(앞선 커밋 0개) PR을 만들 게 없다 — 사용자에게 알리고 멈춘다.
- 현재 브랜치가 베이스 브랜치 자체이면 PR을 만들 수 없다. 작업 브랜치로 옮기도록 안내한다.

## 2. 기존 PR 확인

같은 브랜치로 이미 열린 PR이 있는데 또 만들면 중복이 되므로 먼저 조회한다.

```bash
gh pr list --head "$BRANCH" --state open
```

열린 PR이 있으면 새로 만들지 말고 그 PR을 알리고(`gh pr view --web` 로 열 수 있다), 본문 갱신이 필요한지 사용자에게 묻는다.

## 3. 연결 이슈 식별

작업 브랜치명은 `<type>/<이슈번호>-<slug>` 규칙을 따른다(예: `feature/42-image-search-api`). 브랜치명에서 `<type>/` 다음의 숫자를 이슈 번호로 뽑는다.

```bash
ISSUE=$(git branch --show-current | grep -oE '/[0-9]+' | head -1 | tr -d '/')
```

이슈 번호를 찾으면 제목·본문을 가져온다.

```bash
gh issue view "$ISSUE" --json title,body
```

가져온 이슈의 title 은 **제목**에, body 의 요점은 본문 "문제/신규 기능 설명" 을 쓰는 근거로 삼는다. 이슈 본문을 통째로 복사하지 말고 요점만 옮긴다 — 리뷰어가 길게 읽지 않아도 맥락을 잡게 하는 게 목적이다.

- **브랜치명에 이슈 번호가 없거나 조회에 실패하면**: 사용자에게 연결할 이슈가 있는지 짧게 묻는다. 정말 없으면 제목은 작업 성격을 요약해 쓰고, 본문은 아래 템플릿을 그대로 따른다(`Closes` 줄은 생략).

## 4. 본문 작성 형식

본문은 아래 템플릿을 그대로 따른다. 섹션 순서·명칭을 바꾸지 않는다.

```markdown
## Meta

PR Type: <feature | refactoring | bug-fix | crash-fix 중 하나>

Closes #<이슈번호>

## 문제/신규 기능 설명

<문제나 구현한 기능에 대한 간단한 설명. 버그·크래시면 재현 시나리오를 함께.>

## 적용 버전

<라이브러리 릴리스 버전. 예: 0.0.20>

## 원인

<발생 원인. 신규 기능(feature)이면 "신규 기능 — 해당 없음".>

## 수정/구현

<어떻게 수정·구현했는지. 리뷰어가 중점적으로 봐야 할 주요 포인트를 함께 짚는다.>
```

작성 원칙:

- **섹션 명칭·순서는 위 템플릿 그대로 쓴다** — `Meta` / `문제/신규 기능 설명` / `적용 버전` / `원인` / `수정/구현`.
- **PR Type** 은 커밋 타입·브랜치 type 에서 고른다: `feat`/`feature`→`feature`, `refactor`→`refactoring`, `fix`→`bug-fix`, 크래시 수정→`crash-fix`. 애매하면 사용자에게 확인한다.
- **`Closes #<이슈번호>`** 는 `Meta` 아래에 둔다 — GitHub 가 PR 머지 시 연결 이슈를 자동으로 닫는다. **연결 이슈가 있을 때만** 넣고, 없으면 이 줄을 생략한다. (이게 GitHub 의 표준 이슈 연동 방식이다.)
- **적용 버전** 은 라이브러리 버전(`lib/build.gradle.kts` 의 version, 예: `0.0.20`)에서 끌어온다. 추정이 어려우면 사용자에게 묻는다. 버전과 무관한 변경이면 "해당 없음" 으로 둔다.
- **원인** 은 버그/크래시 수정에만 채운다. 신규 기능이면 "신규 기능 — 해당 없음" 으로 둔다(섹션 자체는 삭제하지 않는다).
- **"문제/신규 기능 설명"·"수정/구현"은 가독성이 전부다.** 파일별로 나열하지 말고 **관심사·기능 단위로 묶고**, 긴 산문 대신 **표와 짧은 불릿**으로 스캔하기 쉽게 쓴다. 이슈 내용을 그대로 되풀이하지 않는다 — 실제 diff·커밋에 근거한다.
- 본문은 한국어로 쓰되, 코드/식별자/명령/경로는 원형 그대로 둔다.
- 자동 생성 푸터(예: "Generated with Claude Code")는 **넣지 않는다.** 본문을 깔끔하게 유지한다.

## 5. 초안 확인

PR을 만들기 전에 **항상** 제목과 본문, 그리고 베이스 브랜치를 사용자에게 보여주고 승인을 받는다.

- **제목**: `<요약>` 형식을 쓴다(예: `이미지 검색 API 연동`). 연결 이슈의 제목(title)을 그대로 쓰면 일관된다. 이슈가 없으면 작업 성격을 요약한 제목을 쓴다. (이슈 연동은 본문 `Closes #N` 으로 하므로 제목에 번호를 강제로 넣지 않아도 된다.)
- **본문**: Meta(PR Type · Closes) / 문제·신규 기능 설명 / 적용 버전 / 원인 / 수정·구현.

사용자가 수정을 요청하면 반영해 다시 보여주고, "생성/올려/좋아" 등 명확한 승인이 있을 때만 다음 단계로 넘어간다. 승인 전에는 **푸시도 PR 생성도 하지 않는다** — 둘 다 원격을 바꾸는 동작이라 되돌리기 번거롭다.

## 6. PR 생성

PR은 브랜치가 원격에 있어야 만들 수 있다. `gh pr create` 는 업스트림이 없으면 푸시 여부를 물으며 만들어 주지만, 비대화형 실행을 위해 먼저 푸시해 두는 것이 안전하다. 1단계의 `git status -sb`에서 푸시되지 않은 커밋(ahead)이 있거나 업스트림이 없으면 먼저 푸시한다.

```bash
# 업스트림이 없으면
git push -u origin "$BRANCH"
# 이미 추적 중인데 ahead면
git push
```

본문에 표·코드블록 등 마크다운이 들어가므로 **본문을 임시 파일에 쓴 뒤 `--body-file` 로 넘긴다.**

```bash
# 본문을 임시 파일로 작성 (Write 도구 사용): /tmp/create-pr-body.md

gh pr create \
  --base "<베이스 브랜치>" \
  --head "<현재 브랜치>" \
  --title "<제목>" \
  --body-file /tmp/create-pr-body.md
```

- `gh pr create` 는 성공 시 PR URL 을 출력한다. 그 URL 을 사용자에게 전달한다.
- 다른 레포(fork 등)로 올리는 경우가 아니면 `--repo` 는 생략한다(현재 레포).

## 예시

브랜치 `feature/42-image-search-api`(이슈 #42 연결), 베이스 `main`.

**제목:** `이미지 검색 API 연동`

```markdown
## Meta

PR Type: feature

Closes #42

## 문제/신규 기능 설명

검색어로 외부 이미지 검색 API를 호출해 결과를 메인 화면 그리드에 표시하는 기능을 추가한다.
스캐폴드 상태이던 앱에서 핵심 가치인 "이미지 검색"이 실제 동작하게 된다.

## 적용 버전

0.0.20

## 원인

신규 기능 — 해당 없음

## 수정/구현

- **네트워크 계층** — `SearchApiService` 인터페이스/구현체를 추가하고 `standardItemsApiHandler` 로 페이징 응답을 매핑.
- **화면** — 검색 결과를 `LazyVerticalGrid` 로 렌더링하고 로딩/빈 상태를 분기.
- 리뷰 포인트: 페이징 토큰 경계 처리(`SearchPagingSource`)와 빈 검색어 가드.
```

"수정/구현"이 파일 목록이 아니라 관심사(네트워크 / 화면)별로 묶이고 리뷰 포인트를 따로 짚은 점에 주목한다 — 리뷰어가 어디를 집중해 볼지 바로 안다. 버그 수정이라면 PR Type 을 `bug-fix` 로 두고 "원인" 에 발생 원인을 채운다.
