# review-commit — 브랜치 커밋 3관점 코드 리뷰

현재 브랜치가 base 브랜치에서 분기한 지점 이후로 쌓은 변경을, 서로 다른 세 관점의
서브에이전트가 **병렬로** 리뷰한다. 한 사람이 모든 관점을 동시에 보긴 어렵기 때문에
관점을 나눠 각자 깊게 파게 하고, 마지막에 한 화면에 모아 보여주는 것이 이 스킬의 목적이다.

## 인자

- `<base>` → 명시한 브랜치/태그/sha 와 HEAD 의 분기점부터 HEAD 까지 (예: `develop`, `master`)
- `<from>..<to>` 또는 `<from> <to>` → 임의 범위 (예: `HEAD~3..HEAD`)

**base 는 반드시 명시한다.** git 은 "현재 브랜치가 어느 브랜치에서 분기했는지" 를 기록하지
않으므로 자동 추정하지 않는다. 인자 없이 호출되면 추측하지 말고 사용자에게 어떤 base 를
기준으로 리뷰할지(예: `develop`/`master`) 되묻는다.

---

## Phase 1 — 리뷰 범위 수집

먼저 헬퍼를 실행해 범위와 아티팩트를 만든다. 인자가 있으면 그대로 넘긴다.

```bash
.claude/skills/review-commit/scripts/collect_diff.sh [인자]
```

출력의 `out_dir:` 경로를 기억한다. 그 디렉터리에 다음 파일이 생성된다:

- `meta.txt` — 범위/통계 요약 (base, range, commit_count, file_count, diffstat)
- `commits.txt` — 범위 내 커밋 목록
- `files.txt` — 변경 파일 목록(name-status)
- `diff.patch` — 전체 통합 diff

탐지된 base·범위·커밋 수를 사용자에게 한 줄로 알리고, 의도와 맞는지 애매하면(예: 커밋이
비정상적으로 많음) 진행 전에 확인한다. 범위가 명확하면 바로 Phase 2 로 간다.

---

## Phase 2 — 3개 서브에이전트 병렬 리뷰

Agent 툴로 아래 3개 에이전트를 **한 메시지에서 동시에** 띄운다(순차로 띄우지 말 것 —
서로 독립적이라 병렬이 빠르다). 셋 다 `general-purpose` 타입으로 충분하다.

세 프롬프트 공통으로 다음을 앞부분에 넣는다:

> 리뷰 범위 아티팩트는 `<out_dir>` 에 있다.
> - `meta.txt`(범위), `commits.txt`(의도 파악용 커밋 메시지), `files.txt`(변경 파일),
>   `diff.patch`(전체 diff) 를 출발점으로 삼아라.
> - diff 가 크면 `diff.patch` 를 통째로 읽기보다 `files.txt` 에서 핵심 파일을 골라 실제
>   파일을 열어 **변경 전후 맥락**까지 보고 판단해라. diff 한 조각만 보고 단정하지 말 것.
> - 작업 의도는 커밋 메시지(및 연결된 GitHub 이슈/PR 제목)에서 유추해 "의도 대비 구현" 으로 본다.

그리고 **공통 출력 규약**을 셋 모두에게 동일하게 준다(아래 "출력 규약" 섹션을 그대로 전달).

각 에이전트의 고유 임무는 다음과 같다.

### 에이전트 A — 지식 베이스(`.claude/docs/`) 준수 리뷰

> 너는 이 프로젝트의 축적된 지식과 변경이 어긋나지 않는지 보는 리뷰어다.
> 먼저 `.claude/docs/` 의 각 카테고리 인덱스(`decisions/`, `conventions/`,
> `troubleshootings/`, `domains/`, `specs/`, `flows/` 의 `README.md`)를 훑어 어떤 항목이
> 있는지 파악하고, **변경과 관련 있는 항목 파일만** 골라 읽어라. 그런 다음 diff 가 각
> 항목을 위반/무시했는지 점검한다:
> - decisions: 이미 내린 기술 결정에 반하는 구현은 없는가 (예: 정해진 패턴 대신 다른 방식).
> - conventions: 네이밍·레이어 의존성·테스트 작성 등 합의된 규칙을 어겼는가.
> - troubleshootings: 과거에 겪고 해결한 함정을 이 변경이 다시 밟고 있지 않은가.
> - domains: 도메인 용어를 코드/네이밍에서 잘못된 의미로 쓰고 있지 않은가.
> - specs/flows: 명세·개발 절차가 정의돼 있다면 그에 어긋나지 않는가.
> 발견 사항마다 **근거가 된 docs 파일 경로**를 반드시 인용한다. 관련 문서가 전혀 없는
> 영역이면 "해당 docs 없음" 으로 솔직히 적고 억지로 만들어내지 않는다.

### 에이전트 B — 사이드 이펙트 / 회귀 위험 리뷰

> 너는 "이 변경이 건드리지 않은 곳을 망가뜨리지 않는가" 를 보는 리뷰어다. 변경 자체의
> 문법보다 **파급(ripple)** 에 집중한다:
> - 변경된 함수/프로퍼티/모델의 **호출부·상속·구현체**를 코드베이스에서 찾아(grep) 이번
>   수정으로 깨지거나 의미가 달라지는 지점이 없는지 본다.
> - 공유 상태·싱글톤·전역 설정·캐시 변경이 다른 화면/기능에 미치는 영향.
> - 라이프사이클(Activity/Fragment/ViewModel), 코루틴 스코프/스레드, 콜백 등록·해제,
>   메모리 누수, 비동기 race 등 타이밍성 부작용.
> - null/경계값/하위호환(직렬화·DB·API 응답 변화)·기존 사용자 데이터 영향.
> "어디가 어떻게 회귀할 수 있는가" 를 구체적인 경로(file:line)와 함께 제시한다.

### 에이전트 C — 숙련 안드로이드 엔지니어 관점 리뷰

> 너는 이 프로젝트를 오래 봐온 시니어 안드로이드 엔지니어다. 동작은 하더라도 더 견고하고
> 관용적인가를 본다:
> - Kotlin 관용성, 불변성, null 안전, 가독성, 불필요한 복잡도.
> - 아키텍처 정합성: core/data/domain/presentation 레이어 경계와 MVVM 패턴 준수, 의존성 방향.
> - Compose/뷰 성능(불필요한 recomposition, 무거운 main-thread 작업), 리소스/문자열 처리.
> - 에러 처리, 로깅, 테스트 가능성·테스트 누락.
> 스타일 취향이 아니라 **유지보수·안정성에 실제로 영향 주는** 지적에 무게를 둔다.

> 세 에이전트 모두 "코드를 새로 짜라" 가 아니라 **리뷰 의견만** 낸다. 거대한 리팩터링
> 요구 대신 변경 범위에 비례한 현실적 지적을 한다.

## 출력 규약 (세 에이전트 공통, 그대로 전달)

각 발견 사항을 아래 형식의 항목으로 모아 **major / minor / suggest** 세 그룹으로 분류해
반환하라. 그룹 안 순서는 영향 큰 것부터.

```
### [major|minor|suggest] <한 줄 제목>
- 위치: <path:line> (해당 시)
- 문제: 무엇이 잘못/우려되는가
- 근거: 왜 문제인가 (에이전트 A 는 인용한 docs 경로 포함)
- 제안: 어떻게 고치면 되는가 (간결히)
```

우선순위 기준:
- **major** — 머지 전 반드시 처리. 버그·크래시·회귀 위험·데이터 손상·결정/컨벤션 명백한 위반.
- **minor** — 고치는 편이 좋음. 사소한 누락, 가독성, 일관성, 경미한 규칙 일탈.
- **suggest** — 선택적 개선 아이디어. 취향/리팩터링 제안.

문제가 없으면 그 에이전트는 "지적 없음" 을 명확히 반환한다. 없는 문제를 만들지 말 것 —
빈 리뷰도 유효한 결과다.

## Phase 3 — 종합 출력

세 에이전트 결과가 모두 오면 한 화면에 보여준다. 리뷰는 관점×심각도라 발견이 많아질수록
쭉 나열하면 한눈에 안 들어온다. 그래서 **위에는 요약 대시보드(분포 + 머지 전 꼭 볼 것),
아래에는 관점별 3개 섹션 상세** 순으로 구성한다 — 읽는 사람이 "전체 규모 → 필수 조치 →
관심 영역 상세" 순으로 자연스럽게 내려가도록.

종합하더라도 각 관점의 지적을 **중복 제거하지 말고** 그대로 살린다(같은 코드를 세 각도에서
본 게 가치다). 단 대시보드의 "머지 전 꼭 볼 것"에서는 동일 major 가 여러 관점에 겹치면
한 번만 싣고 `[📚][💥][🤖]` 처럼 어느 관점들이 짚었는지 태그로 합쳐 보여준다.

### 1) 요약 대시보드 (맨 위)

```markdown
# 🔍 review-commit — <current_branch>
<base>..HEAD · <commit_count> commits · <file_count> files

| 관점 | 🔴 major | 🟡 minor | 🟢 suggest |
|---|:--:|:--:|:--:|
| 📚 docs | 0 | 1 | 1 |
| 💥 side-effect | 0 | 2 | 1 |
| 🤖 android | 1 | 2 | 2 |

🚨 **머지 전 꼭 볼 것 (major N)**
- **[🤖] failure→success 전환으로 '최종 실패' 관측성 상실** — `UpdateStaticResourcesWorker.kt:27`
  → static/reward 워커에 원격추적 로그 추가 + 의도 주석
```

- 표의 숫자는 각 관점이 반환한 그룹별 발견 개수를 그대로 센다.
- "머지 전 꼭 볼 것"은 **major 만** 한 줄 제목 + 위치 + 한 줄 제안으로 압축. major 가 0이면
  이 블록 대신 `✅ **머지 차단급(major) 없음**` 한 줄로 대체한다.

### 2) 관점별 상세 (대시보드 아래)

각 관점을 섹션으로 두고, 섹션 안에서만 major→minor→suggest 로 정렬해 Phase 2 출력 규약의
항목 형식(제목/위치/문제/근거/제안)을 그대로 싣는다.

```markdown
---
## 📚 지식 베이스(docs) 관점
### 🔴 major
_(없음)_
### 🟡 minor
- ...
### 🟢 suggest
- ...

## 💥 사이드 이펙트 관점
...

## 🤖 안드로이드 엔지니어 관점
...
```

규칙:
- 각 섹션에 해당 그룹이 비면 "_(없음)_" 으로 표시해 빠뜨린 게 아님을 분명히 한다.
- 발견이 하나도 없으면 대시보드 표를 0으로 채우고 "머지 전 꼭 볼 것" 자리에
  `✅ **세 관점 모두 지적 없음 — 머지 가능해 보임**` 을 적은 뒤 상세 섹션은 생략한다.
- 파일·라인은 `path:line` 형태로 적어 클릭 가능하게 한다.
- 마지막에 리뷰 범위 아티팩트가 `<out_dir>` 에 있음을 한 줄로 알려, 필요하면 사용자가 직접 볼 수 있게 한다.

> 이 스킬은 리뷰 **의견** 까지만 낸다. 코드 수정은 사용자가 지시할 때 별도로 진행한다.

## Phase 4 — 다음 단계 추천 (AskUserQuestion)

리뷰가 끝난 뒤 추천은 조치할 항목 유무로 갈린다 — 지적 사항을 고쳤으면 그 **수정분을 먼저
커밋**해야 하므로, 곧바로 PR로 가지 않는다.

- **조치할 항목(major/minor)이 있으면**: 먼저 수정을 진행/안내하고(코드 수정은 사용자 지시로
  별도 진행), **수정이 완료된 뒤** `AskUserQuestion` 으로 `create-commit` 을 추천한다. 수정분을
  커밋하면, 그 create-commit 의 다음 단계 추천에서 knowledge-update / create-pr 로 이어진다.
- **조치할 항목이 없으면**: 커밋할 수정분이 없으므로 `AskUserQuestion` 으로 바로 `create-pr` 을
  추천한다.

`AskUserQuestion` 구성:
- 헤더: `다음 단계`
- 수정한 경우 — 질문: "리뷰 지적 사항을 수정했어요. 수정분을 커밋할까요?"
  - **수정분 커밋 (create-commit)** — 수정분을 커밋한다. 고르면 `create-commit` 으로 이어준다.
  - **아니요** — 여기서 멈춘다.
- 수정이 없는 경우 — 질문: "리뷰에서 조치할 항목이 없어요. PR을 생성할까요?"
  - **PR 생성 (create-pr)** — base 브랜치로 향하는 PR을 만든다. 고르면 `create-pr` 로 이어준다(base 를 타겟 브랜치로 넘긴다).
  - **아니요** — 여기서 멈춘다.
