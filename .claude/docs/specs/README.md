# Specs

기능을 설계·구현하기 **전에** "무엇을 만들 것인가"를 정리한 **기능 명세서**를 보관한다.
요구사항·시나리오·범위를 먼저 합의해 구현 중 방향이 흔들리거나 범위가 새는 것을 막는다.
"왜 그렇게 정했는가"는 [decisions](../decisions/), "어떤 순서로 개발하는가"는 [flows](../flows/)에 분리해 기록한다.

## 항목 작성 템플릿

새 항목은 아래 형식으로 `<기능>.md` 파일을 만들어 작성하고, 위 인덱스 표에 한 줄을 추가한다.

```markdown
# <기능 이름> 명세

- 상태: 초안 | 검토 중 | 확정 | 구현 완료
- 갱신일: YYYY-MM-DD
- 관련: (있으면) decisions·flows 항목 링크

## 목적
이 기능이 풀려는 문제와 사용자 가치. 왜 필요한가.

## 사용자 시나리오
- 사용자가 ~할 때 ~할 수 있다.

## 기능 요구사항
1. 무엇을 해야 하는가 — 관찰 가능한 동작 기준으로 기술
2. ...

## 화면 / UX (해당 시)
주요 화면·상태·상호작용.

## 범위 밖 (Out of scope)
이번에 하지 않는 것 — 오해를 막기 위해 명시한다.

## 미해결 질문
- 아직 정해지지 않았거나 결정이 필요한 항목
```
