# Decisions

프로젝트를 진행하며 내려진 **기술 결정 / 개발 방향 결정** 사항을 기록한다.
"왜 이렇게 했는가"를 남겨 같은 논의를 반복하지 않고, 나중에 결정을 재검토할 근거를 제공한다.

## 인덱스

| 파일 | 한 줄 요약 | 상태 | 갱신일 |
|---|---|---|---|
| _(아직 항목 없음)_ | | | |

## 항목 작성 템플릿

새 항목은 아래 형식으로 `<주제>.md` 파일을 만들어 작성하고, 위 인덱스 표에 한 줄을 추가한다.

```markdown
# <결정 제목>

- 날짜: YYYY-MM-DD
- 상태: 제안됨 | 채택됨 | 폐기됨 | 대체됨(→ 대체 항목 링크)

## 맥락
어떤 문제·상황 때문에 결정이 필요했는가.

## 결정
무엇을 하기로 했는가 (한두 문장으로 명확히).

## 근거 / 검토한 대안
왜 이 선택인가. 함께 검토했으나 채택하지 않은 대안과 그 이유.

## 영향
이 결정이 코드/구조/팀 작업 방식에 미치는 영향.
```

## 네이밍 규칙

kebab-case, 결정 주제를 요약. 예: `choose-rest-over-graphql.md`, `single-module-architecture.md`
